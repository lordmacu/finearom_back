Finearom es una empresa colombiana que comercializa fragancias y materias primas para la industria cosmética. El sistema gestiona pedidos (purchase_orders), despachos (partials), cartera y pronósticos de venta (sales_forecasts).

El motor de base de datos es EXCLUSIVAMENTE MariaDB (compatible con protocolo MySQL). Todo el SQL generado debe ser 100% compatible con MariaDB/MySQL — nunca sintaxis de PostgreSQL (NULLS FIRST/LAST, FILTER (WHERE ...), DISTINCT ON, RETURNING, ::cast, ILIKE, generate_series, operadores ->/->>/@>, LATERAL JOIN, string_agg, percentile_cont/disc, date_trunc, EXTRACT(EPOCH FROM …)), ni de SQL Server/T-SQL (TOP N, [corchetes], GETDATE(), DATEPART, ISNULL, CONVERT(date, …), CROSS APPLY/OUTER APPLY, MERGE, IIF, CHARINDEX, LEN), ni de Oracle/PL-SQL (NVL, DUAL implícito, SYSDATE, TO_CHAR/TO_DATE estilo Oracle, CONNECT BY, ROWNUM, DECODE, MINUS), ni ANSI no soportado en MariaDB (FULL JOIN / FULL OUTER JOIN, WITHIN GROUP, window function FILTER clause).

Sintaxis MariaDB equivalente a usar siempre: fechas con NOW(), CURDATE(), DATE_ADD, DATE_SUB, DATEDIFF, DATE_FORMAT, YEAR/MONTH/DAY, STR_TO_DATE; strings con CONCAT, CONCAT_WS, GROUP_CONCAT, SUBSTRING, LOCATE, REPLACE, LIKE; nulls con IFNULL(x, y) o COALESCE(x, y, …) — para nulls al final: ORDER BY col IS NULL, col DESC; paginación con LIMIT n OFFSET m (nunca TOP ni ROWNUM); cast con CAST(x AS CHAR/SIGNED/UNSIGNED/DECIMAL(p,s)/DATE/DATETIME) — nunca x::tipo; window functions ROW_NUMBER, RANK, DENSE_RANK, NTILE, LAG, LEAD, SUM/AVG OVER (soportadas desde MariaDB 10.2+).

FULL JOIN está PROHIBIDO — MariaDB no lo soporta y falla con error 1064. Nunca escribir la palabra FULL en un JOIN. Aplica sobre todo al comparar pronóstico vs real por referencia, donde se quieren todas las referencias (con pronóstico, real o ambos). Para emularlo: arma el conjunto de llaves con UNION (sin ALL, para deduplicar) y luego LEFT JOIN a cada lado, por ejemplo:
```sql
SELECT k.nit, k.codigo,
       COALESCE(p.kilos_pronosticados, 0) AS kilos_pronosticados,
       COALESCE(r.kilos_reales, 0)        AS kilos_reales
FROM (SELECT nit, codigo FROM pronosticos UNION SELECT nit, codigo FROM reales) k
LEFT JOIN pronosticos p ON p.nit = k.nit AND p.codigo = k.codigo
LEFT JOIN reales      r ON r.nit = k.nit AND r.codigo = k.codigo
```

Roles y flujo de las órdenes de compra: las EJECUTIVAS negocian pedidos (clients.executive guarda su email). FRANCY crea la OC (status='pending'), marca is_muestra (orden sin costo) e is_new_win (cliente/producto nuevo). MARLON (logística) revisa las pending, agrega observaciones (new_observation=email al cliente, internal_observation=nota interna), pone fechas estimadas de despacho por producto (partials type='temporal') y la OC pasa a status='processing'; una OC processing con más de 7 días sin despachar está REZAGADA. ALEXA (facturación/despacho) registra los despachos reales cuando la mercancía sale físicamente, creando partials type='real' (dispatch_date, invoice_number, tracking_number, transporter, trm, quantity); la OC pasa a 'parcial_status' (despachó parte) o 'completed' (despachó todo) — una OC puede tener múltiples partials type='real' en cuotas. partials type='temporal' = fecha estimada por Marlon; type='real' = despacho real por Alexa, y es la FUENTE DE VERDAD de lo facturado/despachado. Los estados siguen la secuencia pending→processing→parcial_status→completed; cancelled solo lo puede poner Francy o Marlon y se excluye de las métricas.

Regla absoluta para filtrar por ejecutiva en SQL: cuando el usuario mencione una ejecutiva (nombre completo, apellido, primer nombre, apodo o forma parcial), SIEMPRE hay que resolverla al email canónico del catálogo de ejecutivas (ej. 'Claudia', 'Claudia Cueter', 'María Claudia', 'Cueter' → claudia.cueter@finearom.com). El filtro en SQL debe ser por email, nunca por nombre: usar siempre `c.executive = '<email>'`, nunca `e.name = '...'` ni `c.executive LIKE '...'`. El JOIN a `executives` solo es necesario si se necesita mostrar el nombre completo o el id; si solo se filtra, se puede omitir y usar `c.executive = '<email>'` directo. Si el alias del usuario no resuelve unívocamente a una sola ejecutiva del catálogo, hay que preguntar para desambiguar, nunca inventar un nombre o email.

`JOIN executives e ON e.email = c.executive AND e.is_active = 1` excluye silenciosamente a los clientes asignados a ejecutivas inactivas o sin asignar (unos 33 kilos reales en 2026, 0.3% del total). Si el usuario pregunta POR EJECUTIVA (informe individual, ranking, cumplimiento, comparativa entre ejecutivas) hay que mantener `AND e.is_active = 1`, porque no tiene sentido sumar a personas que ya no están. Si el usuario pregunta por TOTALES GLOBALES del negocio (ventas totales, kilos despachados del año, top productos, participación %), hay que omitir el JOIN a executives y agregar directo sobre partials/clients/sales_forecasts, porque filtrar por is_active=1 perdería esos kilos huérfanos y entregaría cifras incompletas. El indicador para decidir es la columna de agrupación: si el GROUP BY incluye e.id/e.name o el SELECT muestra la ejecutiva, usar el JOIN con is_active=1; si la pregunta es agregada sin desglose por ejecutiva, no hacer el join.

EJECUTIVA — única fuente de verdad es la tabla `executives`: solo son ejecutivas válidas las que existen ahí con is_active=1. La columna clients.executive tiene datos sucios (typos como 'mari.ortega', emails concatenados con coma, strings basura como 'executive', emails genéricos como 'sabores@'); esos clientes no deben salir en reportes por ejecutiva — es el comportamiento correcto que queden fuera.

Collation crítica entre clients.executive (utf8mb4_general_ci) y executives.email (utf8mb4_unicode_ci): cada vez que se compare clients.executive (o cualquier columna derivada de ella dentro de un CTE) con executives.email, es obligatorio usar COLLATE utf8mb4_unicode_ci en uno de los lados, o falla con el error SQL 1267 'Illegal mix of collations'. Aplica en todos los JOINs, igualdades en WHERE, y uniones entre CTEs. El patrón preferido (COLLATE solo una vez) es poner el INNER JOIN a executives dentro de cada CTE que use clients.executive: `INNER JOIN executives e ON e.email = c.executive COLLATE utf8mb4_unicode_ci AND e.is_active = 1` — así los CTEs emergen ya con e.id y e.name y los UNION/LEFT JOIN externos no necesitan COLLATE. El patrón alternativo (executives como tabla base con LEFT JOINs a CTEs) sí necesita COLLATE en el LEFT JOIN final si el CTE trae clients.executive. Aplica en queries sobre purchase_orders, partials y sales_forecasts. Para mostrar el nombre real usar e.name tal cual (con tildes) — nunca SUBSTRING_INDEX/REPLACE del email, deforma los nombres. GROUP BY siempre por e.id, e.name (nunca por el alias).

Mapeo de términos en español a valores reales del campo `status` de purchase_orders: 'pendiente'/'pendientes'/'creada' → 'pending'; 'procesando'/'en proceso'/'procesado'/'en procesamiento' → 'processing'; 'parcial'/'despacho parcial'/'parcialmente despachada' → 'parcial_status'; 'completada'/'completa'/'entregada'/'despachada' → 'completed'; 'cancelada'/'cancelado'/'anulada' → 'cancelled'.

Regla crítica sobre cuándo NO filtrar por fecha de creación: el filtro order_creation_date BETWEEN solo aplica cuando el usuario pregunta por órdenes CREADAS en un período específico. Nunca usar order_creation_date BETWEEN cuando el filtro principal es por status, porque las órdenes activas pueden ser de cualquier fecha (ej. WHERE po.status = 'processing', 'parcial_status', IN ('processing','parcial_status'), o 'pending' → sin filtro de fecha de creación). Para órdenes 'estancadas'/'sin despacho'/'sin movimiento' filtrar por status + DATEDIFF(CURDATE(), po.order_creation_date) > X. El pipeline (órdenes pending/processing/parcial_status sin completar) tampoco lleva filtro de fecha de creación.

Cómo saber qué queda pendiente en una OC en parcial_status: kilos pedidos = SUM(pop.quantity) de purchase_order_product WHERE muestra=0; kilos despachados = SUM(par.quantity) de partials WHERE type='real' AND deleted_at IS NULL; kilos pendientes = kilos_pedidos - kilos_despachados (si es mayor a 0 hay mercancía sin despachar). Una OC pasa a 'completed' cuando Alexa despacha el resto y no quedan kilos pendientes.

Comportamiento crítico de partials — siempre soft delete + reemplazo: cuando Alexa registra o actualiza un despacho, el sistema borra (soft delete) TODOS los partials type='real' existentes de esa OC y crea los nuevos desde cero (los anteriores quedan con deleted_at != NULL, son historial, no despachos vigentes). Cuando Marlon actualiza fechas estimadas, el sistema borra (soft delete) todos los partials type='temporal' y crea los nuevos, con el mismo patrón exacto. Regla absoluta: en cualquier query sobre partials siempre incluir `AND par.deleted_at IS NULL`.

Inflación de kilos pedidos — regla crítica: nunca calcular SUM(pop.quantity) en el mismo CTE que itera sobre partials, porque una línea de OC puede tener N partials reales (despachos en cuotas) y pop.quantity se contaría N veces. El patrón correcto usa dos CTEs completamente independientes: uno que arranca desde partials y calcula SOLO kilos/valor despachados (SUM(par.quantity)), y otro que arranca desde purchase_order_product y calcula SOLO kilos pedidos (SUM(pop.quantity)), filtrando por OCs que tuvieron despachos usando EXISTS; al final se unen los dos CTEs por client_id o el nivel de agrupación requerido. Este patrón elimina la inflación porque pop nunca se une directamente a partials.

Métricas clave del dashboard: "Órdenes DESPACHADAS en el período" = OCs con al menos un partial type='real' y dispatch_date en el período (usar EXISTS o JOIN por partials.dispatch_date, siempre con par.type='real' AND par.dispatch_date BETWEEN fechas AND par.deleted_at IS NULL AND pop.muestra=0; nunca usar order_creation_date para esto). "Órdenes CREADAS en el período" = OCs donde order_creation_date BETWEEN fechas, filtrando directo en purchase_orders sin usar partials.

Patrón correcto para "stats por ejecutiva (pedido vs despachado)": empezar siempre desde partials con dispatch_date en el período; SUM(pop.quantity) = kilos pedidos (denominador), SUM(par.quantity) = kilos despachados (numerador); agrupar por c.executive; nunca filtrar por order_creation_date porque se perderían ejecutivas con OCs creadas antes del período.

"Valor total OC" = SUM(pop.quantity * precio), usando pop.quantity (kilos PEDIDOS) para el denominador del cumplimiento — es la excepción a la regla general de usar par.quantity; siempre necesita JOIN products p ON pop.product_id=p.id. "Despachado/Facturado" = SUM(par.quantity * precio), usando par.quantity (kilos REALES DESPACHADOS). "Cumplimiento %" = despachado_usd / total_oc_usd × 100, calculado desde partials (no desde order_creation_date). "Fill Rate" = SUM(par.quantity) / SUM(pop.quantity) × 100 en kilos, mide qué porcentaje de lo pedido se despachó.

"Valor COP" = valor_usd × TRM normalizada, con cascada: partial.trm → po.trm → trm_daily → 4000 (el +0 es necesario porque trm es string en MariaDB). Normalización TRM obligatoria: algunos partials tienen la trm almacenada ×100 (ej. 370005 en vez de 3700.05), así que siempre hay que validar el rango: `CASE WHEN NULLIF(par.trm+0,0) BETWEEN 3200 AND 10000 THEN NULLIF(par.trm+0,0) WHEN NULLIF(par.trm+0,0) > 10000 THEN NULLIF(par.trm+0,0)/100 ELSE NULL END`, aplicando el mismo patrón a po.trm, con fallback final a trm_daily → 4000.

Crítico ONLY_FULL_GROUP_BY + TRM diaria: la subconsulta correlacionada a trm_daily (…WHERE td.date <= par.dispatch_date…) referencia par.dispatch_date; si se deja dentro de un SUM()/agregado en una query con GROUP BY, MariaDB lanza el error 1055 "dispatch_date isn't in GROUP BY" — es un falso positivo del analizador, la cascada es correcta pero MariaDB no la acepta a través del borde de la subconsulta. Solución obligatoria cuando la query agrupa (por cliente/ejecutiva/producto/mes): calcular valor_usd y valor_cop POR FILA en una subconsulta derivada SIN GROUP BY, y hacer el SUM + GROUP BY en la consulta externa; nunca poner la subconsulta a trm_daily con par.dispatch_date dentro de un SUM agrupado. Si la query NO agrupa (detalle fila por fila), sí se puede usar la cascada TRM inline sin subconsulta derivada — la subconsulta derivada solo es obligatoria cuando hay GROUP BY.

"Kilos despachados" = SUM(par.quantity) de partials reales del período (sin muestras). "Pipeline / Pendiente por facturar" = SUM de partials type='temporal' activos (deleted_at IS NULL), valor aún no despachado pero programado. "New Win (nivel OC)" = COUNT(DISTINCT CASE WHEN po.is_new_win=1 THEN po.id END) — nunca SUM(is_new_win) sin DISTINCT, porque contaría líneas y no órdenes. "New Win rate" = new_win_orders / total_orders_creadas × 100. "New Win (nivel línea)" = COUNT(DISTINCT CASE WHEN pop.new_win=1 THEN pop.id END). "Días de atraso vs fecha estimada" = GREATEST(DATEDIFF(CURDATE(), fecha_estimada), 0), siempre usando GREATEST para evitar valores negativos cuando la fecha estimada es futura. "Muestra (nivel OC)" = po.is_muestra=1 (toda la OC es muestra, excluir de totales); "Muestra (nivel línea)" = pop.muestra=1 (solo esa línea es muestra); en queries de valor/kilos siempre filtrar AND pop.muestra=0. "Órdenes rezagadas" = po.status='processing' AND order_creation_date <= CURDATE() - 7 días (aprobadas por Marlon pero Alexa aún no despacha). "Planeado/pendiente del período" = OCs con fecha de despacho estimada en el rango, donde fecha de despacho = COALESCE(MIN(partials_temporal.dispatch_date), pop.delivery_date, po.dispatch_date). Regla: si el usuario dice 'creadas' usar po.order_creation_date; si dice 'pendientes'/'planeadas'/'con despacho en'/'del dashboard' usar esa cascada COALESCE; si dice 'despachadas'/'facturadas'/'enviadas' usar partials.dispatch_date con type='real' AND deleted_at IS NULL.

Lead time y tipos de cliente: client_type tiene DOS clasificaciones distintas — la comercial 'AA' (mayor prioridad) > 'A' > 'B' > 'C' (menor prioridad), usada para lead time, y la de portafolio legado 'pareto' (estratégico)/'balance'/'none'. clients.lead_time (días) es el tiempo de entrega estándar del cliente (AA/A = 9 días, C = 12 días). El lead time real de una OC = DATEDIFF(fecha_primer_despacho_real, po.order_creation_date). On-time significa que el despacho real ocurrió en o antes de la fecha acordada de entrega: siempre usar pop.delivery_date (en purchase_order_product), la fecha acordada por línea, la más precisa para on-time — po.required_delivery_date es la fecha que pidió originalmente el cliente y suele ser anterior al despacho real, no usar para on-time. Nunca usar pop.required_delivery_date: ese campo NO existe en purchase_order_product. Para análisis de lead time usar JOIN con c.client_type IN ('AA','A','B','C').

Cartera — formato y lógica: saldo_contable y saldo_vencido son strings con PUNTO como separador decimal (ej. '26857379.12', '-256309.65'), ya son decimales normales. Para operar numéricamente usar CAST(saldo_contable AS DECIMAL(15,2)) sin REPLACE, porque el punto ya es el decimal; nunca usar REPLACE para quitar puntos, destruye el separador decimal y multiplica el valor por 10. Para ordenar: ORDER BY CAST(saldo_contable AS DECIMAL(15,2)) DESC.

En cartera, el campo `dias` es positivo cuando la factura está por vencer y negativo cuando está VENCIDA hace |dias| días. Las facturas vencidas se filtran con WHERE dias < 0, y para días de mora se usa ABS(ca.dias) — nunca DATEDIFF(CURDATE(), fecha_cartera). Siempre hay que filtrar por el ÚLTIMO SNAPSHOT COMPLETO (la vista actual de Siigo), no por el MAX(fecha_cartera) global: `fecha_cartera = (SELECT fecha_cartera FROM cartera GROUP BY fecha_cartera HAVING COUNT(DISTINCT nit) >= 0.5 * (SELECT MAX(cnt) FROM (SELECT COUNT(DISTINCT nit) cnt FROM cartera GROUP BY fecha_cartera) z) ORDER BY fecha_cartera DESC LIMIT 1)`. Sin filtrar por snapshot se traen todos los cortes históricos y se repiten miles de filas. cartera.documento (número de factura) se cruza con recaudos.numero_factura para ver pagos. La deuda neta real es saldo_contable menos lo pagado: MAX(saldo - SUM(recaudos.valor_cancelado), 0). catera_type puede ser 'nacional' o 'internacional' para filtrar cartera de exportación vs local.

Cartera.vendedor y cartera.nombre_vendedor son el vendedor en SIIGO (ERP) y pueden NO coincidir con clients.executive — para filtrar cartera por ejecutiva hay que hacer JOIN cartera ca ON ca.nit = clients.nit y filtrar por clients.executive, nunca por ca.vendedor. recaudos.nit es BIGINT (número sin texto) mientras clients.nit y cartera.nit son VARCHAR; el JOIN funciona con conversión implícita en MariaDB.

La tabla cartera tiene UNA FILA POR FACTURA, no por cliente: si se agrupa por nit/cliente hay que usar SUM(CAST(saldo_contable AS DECIMAL(15,2))), nunca saldo_contable directo en el SELECT; si se muestra detalle por factura (sin GROUP BY), sí se puede usar el CAST directo. En MariaDB con ONLY_FULL_GROUP_BY, todo campo no agregado en el SELECT debe estar en el GROUP BY. Nunca hacer JOIN directo de cartera a una query que ya tiene GROUP BY de despachos.

Cartera vencida en un scorecard por ejecutiva debe incluir TODOS los clientes de la ejecutiva, no solo los que tuvieron despachos en el período — un cliente puede tener cartera vencida aunque no haya despachado en el mes en cuestión; si se filtra por partials, esos clientes quedan fuera y la cartera aparece como 0. Para cartera vencida por ejecutiva siempre usar una subconsulta correlacionada, nunca un CTE que agrupe por c.executive (ese campo es sucio y no hace match con executives.email): cuando la base del query es clients (c), usar `AND cc.executive = c.executive`; cuando la base es executives (e), usar `AND cc.executive COLLATE utf8mb4_unicode_ci = e.email`. Nunca usar SUM(DISTINCT ca.saldo_vencido) ni LEFT JOIN de cartera al flujo principal de despachos para calcular cartera por ejecutiva.

Regla crítica de cantidades: cuando se hace JOIN partials → purchase_order_product, siempre usar par.quantity para calcular el valor despachado. pop.quantity es la cantidad PEDIDA; par.quantity es la cantidad REAL DESPACHADA.

Joins especiales: la ciudad de entrega o sucursal se obtiene con LEFT JOIN branch_offices bo ON pop.branch_office_id = bo.id, usando bo.delivery_city y bo.name. Las guías del período se obtienen con SELECT DISTINCT tracking_number, transporter FROM partials WHERE type='real' AND deleted_at IS NULL AND dispatch_date BETWEEN fechas. Las facturas del período se obtienen con SELECT DISTINCT invoice_number FROM partials con el mismo filtro.

Pronósticos de ventas (tabla sales_forecasts) — para preguntas de "pronosticado vs real", "cumplimiento", "presupuesto", "forecast" o "lo proyectado": una fila es un pronóstico de kilos por cliente + producto + año + mes + modelo (modelo ∈ xgboost|theta|croston|holt_winters|manual; año es VARCHAR de 4 caracteres; mes es VARCHAR en español mayúsculas: 'ENERO' a 'DICIEMBRE'). Cuando el usuario pregunta "pronosticado", "presupuesto", "plan" o "lo que se esperaba vender" sin mencionar modelo, usar siempre modelo='manual' (es el presupuesto que carga el equipo comercial); los otros modelos son predicciones estadísticas y solo se usan si el usuario las pide por nombre.

Collation y duplicados en forecasts: products.code es utf8mb4_general_ci y sales_forecasts.codigo es utf8mb4_unicode_ci; products.code NO es único globalmente (el mismo código puede existir para varios clientes, y a veces duplicado dentro del mismo cliente), así que nunca se debe unir forecasts con products usando solo p.code = sf.codigo porque duplicaría kilos y valores. El join mínimo correcto es JOIN clients c ON c.nit = sf.nit y luego JOIN products p ON p.client_id = c.id AND p.code COLLATE utf8mb4_unicode_ci = sf.codigo. Si el query calcula valores agregados, usar un CTE `product_ranked` con ROW_NUMBER() PARTITION BY p.client_id, p.code y filtrar rn=1 para asegurar una sola fila de precio por cliente+código antes de sumar. clients.nit = sales_forecasts.nit no requiere COLLATE (ambos son unicode_ci).

sf.año es VARCHAR: comparar con literal entre comillas ('2026') o CAST(sf.año AS UNSIGNED) = 2026; para cruzar con YEAR(par.dispatch_date) usar sf.año = CAST(YEAR(par.dispatch_date) AS CHAR). sf.mes es VARCHAR en español: para cruzar con MONTH(par.dispatch_date) usar FIELD: `MONTH(par.dispatch_date) = FIELD(sf.mes,'ENERO','FEBRERO','MARZO','ABRIL','MAYO','JUNIO','JULIO','AGOSTO','SEPTIEMBRE','OCTUBRE','NOVIEMBRE','DICIEMBRE')`.

Para "total pronosticado por cliente" o "total del presupuesto por cliente", agrupar SOLO por c.client_name, c.nit — no incluir referencia/producto/código en SELECT ni GROUP BY salvo que el usuario pida explícitamente "por referencia", "por producto", "por código" o "desglose". Si piden "en pesos"/"COP"/"valor en pesos" para forecasts, calcular valor_usd = SUM(cantidad_forecast * precio) y valor_cop = valor_usd × TRM más reciente: COALESCE((SELECT value FROM trm_daily WHERE date <= CURDATE() ORDER BY date DESC LIMIT 1), 4000), porque los forecasts no tienen TRM propia.

Regla absoluta anti-duplicación de forecasts: la tabla products tiene duplicados reales por (client_id, code) — el mismo código puede aparecer 2-3 veces con el mismo o distinto product_name. Si se hace `JOIN products p ON p.client_id=c.id AND p.code=sf.codigo` sin deduplicar, cada forecast se multiplica por el número de duplicados (caso confirmado: Jabones El Tigre, code 14400, tiene 2 filas en products, y SUM(cantidad_forecast) reportaba 1000 cuando el real eran 500). Esto no se detecta visualmente porque GROUP BY product_name colapsa filas idénticas pero el SUM ya quedó duplicado. Siempre que la query sume/agregue sf.cantidad_forecast y haga JOIN a products, usar primero el CTE product_ranked y filtrar p.rn = 1 en el join al outer.

Anomalía de datos (forecasts huérfanos): existen unas 86 filas en sales_forecasts (modelo=manual, 2026) cuyo código no matchea ningún product para ese cliente (≈13,834 kg, 6.8% del presupuesto anual). Queries con JOIN product_ranked p ... AND p.code = sf.codigo los excluyen silenciosamente — es aceptable para análisis "por producto" porque no se puede mostrar un producto que no existe. Queries agregadas a nivel cliente (sin desglose por producto) sí los incluyen porque no joinean products; por eso los totales de presupuesto del año difieren entre ambos tipos de query. Si el usuario nota la diferencia, explicar que hay forecasts con códigos no asignados al catálogo del cliente.

Anomalía de datos (OCs completed con pendientes): existen 711 OCs en status='completed' con pop.quantity > sum(partials.real.quantity) (≈67K kg). Para "kilos pendientes" o "qué falta despachar", no filtrar solo por status='parcial_status'; usar status IN ('processing', 'parcial_status', 'completed') y filtrar por pop.quantity > COALESCE(kilos_reales, 0).

Patrón recomendado para real vs pronóstico por producto y año: usar un CTE product_ranked (ROW_NUMBER PARTITION BY client_id, code), un CTE `reales` que agrega SUM(par.quantity) por c.nit, p.code, YEAR(par.dispatch_date) filtrando par.type='real' AND par.deleted_at IS NULL AND pop.muestra=0, y luego el SELECT final desde sales_forecasts con JOIN clients, JOIN product_ranked (rn=1) y LEFT JOIN reales, filtrando sf.modelo='manual'. product_ranked no afecta el CTE `reales` (los partials apuntan a un product_id específico, no se duplican); solo se usa en el join externo para que el código del forecast empate con una sola fila de products. Si la query no suma cantidad_forecast (ej. solo cuenta clientes distintos), el join simple a products es seguro, pero ante cualquier SUM(sf.cantidad_forecast) o SUM(sf.cantidad_forecast * price) hay que usar product_ranked. En el CTE `reales` siempre filtrar po.status != 'cancelled' — los partials de OCs canceladas no cuentan como despacho real.

Interpretación de intención en preguntas de forecast: "qué falta por vender"/"qué no ha salido"/"pendientes"/"brechas" pregunta por productos donde lo despachado es MENOR a lo pronosticado (no solo cero) — filtrar WHERE pronosticado > COALESCE(real, 0), no real = 0, e incluir kilos_faltantes y cumplimiento_pct, ordenando por kilos_faltantes DESC. "Sin despachar nada"/"no se ha despachado en absoluto"/"cero ventas" sí filtra COALESCE(real,0) = 0 porque el usuario pidió específicamente los que no han salido del todo. Si piden por mes, agregar sf.mes en SELECT/GROUP BY y cruzar con FIELD(sf.mes,...) en el LEFT JOIN.

Alinear el período entre pronóstico y real es un error frecuente y grave: ambos DEBEN cubrir exactamente los mismos meses. Si el usuario pide un rango parcial ("los N meses cerrados", "enero a mayo", "lo que va del año", "primer trimestre"), filtrar los dos lados a ese rango — reales con par.dispatch_date BETWEEN '<ini>' AND '<fin>', y pronóstico con sf.año='<año>' AND sf.mes IN (...) con exactamente esos mismos meses. Nunca dividir el despacho de N meses entre el pronóstico de los 12 meses, porque el % de cumplimiento sale falsamente bajo (caso real: Quimisense enero-mayo daba 32% con 5 meses de real ÷ 12 de pronóstico; alineando ambos lados a enero-mayo da 82%). Antes de entregar la query, verificar que el filtro de sf.mes y el rango de par.dispatch_date representen el mismo conjunto de meses.

Para no perder forecasts huérfanos en reportes por referencia, usar LEFT JOIN product_ranked (no INNER JOIN): con INNER, los códigos pronosticados sin fila de producto en el catálogo del cliente desaparecen y el total pronosticado queda corto; con LEFT JOIN la referencia aparece (usando COALESCE(p.product_name, sf.codigo) para el nombre) y su pronóstico sí cuenta.

Alcance del reporte de forecasts: si es GENERAL (todos los clientes con presupuesto), no filtrar por c.nit y agrupar por cliente y/o producto; si es POR CLIENTE, añadir WHERE c.client_name LIKE '%NOMBRE%' o c.nit='...'; si es POR EJECUTIVA, WHERE c.executive = 'email' y agrupar por ejecutiva.

Crítico real vs pronóstico — no sesgar hacia un lado: un LEFT JOIN 'FROM creadas LEFT JOIN pronosticado' oculta ejecutivas/clientes con presupuesto pero sin ventas (0% de cumplimiento, que es lo más importante de detectar) — nunca hacer eso. El patrón correcto construye un CTE `claves` con UNION de ambos lados y hace dos LEFT JOINs: `WITH creadas AS (...), pronosticado AS (...), claves AS (SELECT ejecutiva_email FROM creadas UNION SELECT ejecutiva_email FROM pronosticado) SELECT k.ejecutiva_email, COALESCE(cr.kilos_pedidos,0), COALESCE(pr.kilos_pronosticados,0), ... FROM claves k LEFT JOIN creadas cr USING(ejecutiva_email) LEFT JOIN pronosticado pr USING(ejecutiva_email)`. El cumplimiento debe manejar NULL en ambos lados: si no hay forecast, cumplimiento='sin presupuesto'; si no hay ventas, cumplimiento=0%. El filtro en 'creadas' siempre debe llevar AND po.status != 'cancelled' para no inflar con órdenes anuladas.

Reglas SQL generales: usar siempre las fechas del período activo en los filtros. Aliases recomendados: client_name, ejecutiva, kilos, valor_usd, valor_cop, ocs, fill_rate_pct, pipeline_usd, fecha_despacho, numero_oc. Con MariaDB en modo ONLY_FULL_GROUP_BY, todos los campos no agregados del SELECT deben estar en el GROUP BY, agrupando por c.executive (no por el alias); cuando se use po.order_consecutive, c.nit, c.executive o cualquier campo de po/c en el SELECT junto con GROUP BY, hay que incluir esos campos en el GROUP BY (ej. GROUP BY po.id, po.order_consecutive, c.client_name, c.nit, c.executive, po.status, po.order_creation_date).

Precio en GROUP BY: pop.price y p.price nunca pueden ir fuera de un SUM/AVG/MAX cuando la query tiene GROUP BY a nivel de orden/cliente/ejecutiva, porque MariaDB rompe con "isn't in GROUP BY" — el precio es por renglón de purchase_order_product, no por orden. Mal: `ROUND((SUM(pop.quantity) - COALESCE(SUM(par.quantity),0)) * COALESCE(NULLIF(pop.price,0), p.price, 0), 2)`. Bien: `ROUND(SUM((pop.quantity - COALESCE(par.quantity,0)) * COALESCE(NULLIF(pop.price,0), p.price, 0)), 2)` — la regla general es mover la multiplicación de cantidad × precio DENTRO del SUM, nunca hacer SUM(cantidad) * precio sino siempre SUM(cantidad * precio).

Cuando el usuario pida el desglose "por referencia", "por producto", "por código", "por SKU", "qué referencias compró" o "cuánto de cada referencia/producto", es obligatorio incluir p.id, p.code, p.product_name en el SELECT y en el GROUP BY junto con el resto de dimensiones (año/mes/cliente) — no basta con agrupar solo por año, por ejemplo. El alias recomendado para el nombre del producto es 'referencia'. Nunca incluir columnas id en el SELECT final; siempre incluir el NIT junto al nombre del cliente.

Clientes nuevos: cuando el usuario pida "clientes nuevos", "first order este año/mes" o "clientes que entraron este año", nunca usar HAVING MIN(po.order_creation_date) >= fecha sobre OCs ya filtradas por status, porque un cliente con órdenes completadas en años anteriores pasaría el filtro (sus OCs viejas no tienen status activo). El patrón correcto verifica la primera orden en TODA la historia del cliente con una subconsulta: `HAVING (SELECT MIN(po_all.order_creation_date) FROM purchase_orders po_all WHERE po_all.client_id = c.id) >= 'FECHA_INICIO'`, garantizando que el cliente no tiene ninguna orden anterior a esa fecha en todo el sistema.

GROUP_CONCAT para comparar pertenencia entre conjuntos: nunca usar GROUP_CONCAT junto con IS NULL/NOT NULL para detectar si un elemento específico estuvo en un período anterior, porque GROUP_CONCAT devuelve un string y el IS NULL solo verifica si el grupo existe, no si el elemento está dentro (esto da 'recurrente' a todos cuando el mes anterior existe). El patrón correcto usa EXISTS: `EXISTS (SELECT 1 FROM clientes_por_mes prev WHERE prev.client_id = cpm.client_id AND prev.mes_num = cpm.mes_num - 1)`. Aplica a retención de clientes, clientes nuevos vs recurrentes, y cualquier comparación de sets entre períodos.

PERCENTILE_CONT / PERCENTILE_DISC no existen en MariaDB. Para percentiles usar NTILE(N) OVER (ORDER BY col ASC/DESC): top 20% (percentil 80) es NTILE(5) OVER (ORDER BY valor ASC) = 5; la mediana (percentil 50) es NTILE(2) OVER (ORDER BY valor ASC) = 2. MariaDB tampoco soporta NULLS LAST / NULLS FIRST (sintaxis Postgres/Oracle): para poner nulls al final en ORDER BY DESC usar ORDER BY col IS NULL, col DESC (o col IS NULL, col ASC para ascendente).

Para combinar dos conjuntos donde ambos pueden tener filas sin match (equivalente a FULL OUTER JOIN, no soportado), el patrón correcto es un CTE adicional con UNION de ambos lados y dos LEFT JOINs: `WITH cte_a AS (...), cte_b AS (...), todos AS (SELECT key FROM cte_a UNION SELECT key FROM cte_b) SELECT ... FROM todos LEFT JOIN cte_a ON ... LEFT JOIN cte_b ON ...`. Aplica siempre que se necesite comparar dos períodos (año actual vs año anterior, mes vs mes) por ejecutiva, cliente, producto, etc.

CTEs que deben devolver una fila por clave: si un CTE obtiene un valor único por entidad (ej. TRM por OC, última fecha por cliente) y luego se hace JOIN, el CTE debe tener GROUP BY en la clave, porque sin GROUP BY devuelve todas las filas y el JOIN multiplica los resultados. Mal: `SELECT order_id, trm FROM partials WHERE ... ORDER BY dispatch_date DESC` (devuelve N filas por order_id). Bien: `SELECT order_id, MAX(NULLIF(trm+0,0)) AS trm FROM partials WHERE ... GROUP BY order_id`. Aplica a TRM por OC, última fecha por cliente/OC, y cualquier "último valor" por entidad.

WITH ROLLUP — regla crítica: el GROUP BY debe tener exactamente UNA columna cuando se usa WITH ROLLUP para obtener solo el subtotal general, porque cada columna extra en el GROUP BY genera N niveles adicionales de subtotales intermedios y produce filas duplicadas. Mal: `GROUP BY pp.executive, ea.nombre_real, cv.clientes_con_vencido, tc.total_clientes WITH ROLLUP` (produce 17 filas para 4 ejecutivas). Bien: `GROUP BY pp.executive WITH ROLLUP` (produce exactamente 5 filas: 4 ejecutivas + 1 TOTAL); las otras columnas van con MAX() o ANY_VALUE() en el SELECT, y la fila del TOTAL usa COALESCE(MAX(ea.nombre_real), 'TOTAL') AS ejecutiva.

Proceso de orden (timeline): cuando el usuario mencione una GUÍA (10 dígitos) o una OC (formato YYYY-..., ej. 2327-2699), generar el SQL de timeline con UNION ALL usando una columna fase_orden para ordenar: real(3) primero, estimado(2), creación(1) al final. Usar c.client_name (no c.name). El estado se infiere: creación='pending', estimado='processing', real=po.status actual. Por guía, filtrar con tracking_number='{NUMERO}' en las subqueries; por OC, filtrar con (SELECT id FROM purchase_orders WHERE order_consecutive='{OC}' LIMIT 1). El resultado se ordena por ORDER BY fase_orden DESC, fecha DESC, mostrando fase, fecha, OC, cliente, estado, factura, guía, kilos, y disponiendo transportador, NIT y ejecutiva como campos adicionales.

Hallazgo verificado contra datos de producción: los estados reales de purchase_orders son 'completed', 'processing', 'parcial_status' y 'cancelled'. NO existe el estado 'pending' en los datos. Una orden "esperando despacho" se identifica por ausencia de parciales type='real' (o type='temporal' para "esperando la fecha estimada de Marlon"), no por status='pending'. Excluir siempre 'cancelled' de las métricas de venta.

Hallazgo verificado de cartera: la tabla cartera recibe cada lunes un sync COMPLETO desde Siigo (~33-35 clientes = todos los que hoy tienen cartera) y además imports ad-hoc de 1-6 clientes en otros días. La "cartera actual" = la VISTA ACTUAL DE SIIGO = el ÚLTIMO SNAPSHOT COMPLETO. Se detecta por umbral RELATIVO de # de clientes distintos (la fecha más reciente cuyo # de nits distintos es al menos el 50% del máximo # de nits visto en cualquier snapshot): `car.fecha_cartera = (SELECT fecha_cartera FROM cartera GROUP BY fecha_cartera HAVING COUNT(DISTINCT nit) >= 0.5 * (SELECT MAX(cnt) FROM (SELECT COUNT(DISTINCT nit) cnt FROM cartera GROUP BY fecha_cartera) z) ORDER BY fecha_cartera DESC LIMIT 1)`. NUNCA uses el MAX(fecha_cartera) GLOBAL: a veces el corte más reciente es un ad-hoc de 1 cliente y devuelve resultados casi vacíos. NUNCA correlaciones el snapshot por-nit (MAX por cada nit): eso resucita saldos viejos de clientes que ya pagaron y salieron de Siigo, SOBRECONTANDO la cartera. Un cliente ausente del último snapshot completo = ya pagó = NO cuenta.

Hallazgo verificado: cartera.saldo_contable y cartera.saldo_vencido están en COP (pesos), no en USD, y son decimales puros (punto), así que CAST(saldo_contable AS DECIMAL(20,2)) directo es correcto. Cartera vencida = car.dias < 0. Saldo a favor = saldo neto negativo.

Hallazgo verificado: la tabla recaudos cruza con cartera por factura de forma imperfecta porque cartera.documento trae sufijos (ej. 'F-003-...-001') que recaudos.numero_factura no tiene; para cruzarlos usa SUBSTRING_INDEX(car.documento,'-',3) o LIKE CONCAT(numero_factura,'%'), o agrega por nit. Los datos de recaudos son acotados en el tiempo (verifica el rango antes de filtrar por fecha).

Hallazgo verificado: clients.executive guarda un email pero hay valores sucios (typos como 'mari.ortega@', huérfanos como 'sabores@finearom.com'). Para resolver o mostrar ejecutiva, cruza contra la tabla executives (email, is_active=1) usando COLLATE utf8mb4_unicode_ci (o general_ci) para evitar 'Illegal mix of collations'. Solo hay 4 ejecutivas activas. Todos los parciales reales tienen invoice_number (no hay despachos "sin factura"); sí hay despachos sin tracking_number (sin guía).
